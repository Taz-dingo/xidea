# Spec

## Goal

用最小可演示的方式证明 Xidea 的核心差异点：
它不是单一学习工具，而是一个按项目组织的 AI 学习编排系统，能够接住多样输入，并根据学习者状态和项目上下文选择不同训练动作。

长期目标上，Xidea 要同时成立三件事：

- 支持多模态输入，而不是只接文本问答
- 支持多类型学习形式，由系统决定当前更适合哪一种
- 具备类似 Anki 的科学复习系统，用独立的 SRS / Review Engine 决定何时复习与复习什么

当前比赛版默认只围绕一个主案例展开：
用户正在推进一个 AI 应用项目，需要真正理解并应用 RAG 系统设计，而不只是临时问答或被动看资料。

这只是比赛版的证明路径，不是长期产品边界。
长期上，Xidea 仍然面向更广义的项目型学习任务，只是当前先用一个最容易讲清楚的主案例来证明系统价值。

## In Scope

- 锁定一个主案例来承载比赛叙事：AI 工程师围绕真实项目学习 RAG 系统设计
- 用 mock 数据表达同一线程里的普通问答与随时附加材料
- 用 `project / thread` 组织学习上下文、材料、问题历史和训练历史
- 用显式的 `LearnerState` 同时表达理解状态和记忆状态
- 用受约束的单 agent 表达系统如何选择训练动作
- 允许同一系统支持多种学习形式，但比赛版只先挑 2 到 3 种最能讲清主线的动作
- 区分 `Review Engine`、`Agent Memory` 和 `Consolidation` 三层能力
- 用 React 前端展示学习线程、动作反馈、右侧状态 inspector 和最小必要的编排可见性
- 第一版优先讲清楚 why this action now，而不是把某一种学习形式做成产品本体
- 每个 demo 输出都要让人理解：输入上下文 -> 学习状态 -> 动作选择 -> 状态回写 这条链路确实存在，但用户主线程优先看到的是动作与反馈，而不是内部编排字段平铺
- 用 LangChain + LangGraph 承接第一版 agent runtime，但只启用最小必要能力
- 用仓库内文档支撑三人协作和 agent 上下文接力

## Out Of Scope

- 面向所有学科和所有人群的广义学习平台叙事
- 没有项目目标、没有持续上下文的泛化陪聊式 tutor
- 真实多模态解析链路
- 真实语音、视频理解
- 完整 spaced repetition 引擎
- 大范围页面扩张
- 开放式 autonomous agent
- 为了“看起来完整”而增加的非关键功能

## Core Demo Story

1. 用户围绕一个正在推进的 AI 项目进入系统，当前目标是学会并应用 RAG 系统设计
2. 用户在同一个 `project / thread` 里既可以直接提问，也可以随时补充原始材料；材料不再是单独入口模式
3. 系统一边读取项目上下文，一边形成学习者画像、当前状态判断和可复习对象
4. 系统基于状态决定更适合的训练方式，如概念澄清、导师问答、主动回忆、情境练习或科学复习
5. 系统按判断选择是否发起一张或一组顺序学习动作卡，并在作答后给出短反馈，再决定下一步
6. 页面必须把“系统看到了什么、为什么是这个动作、作答后如何更新状态”讲清楚；用户主线程优先看到动作与反馈，状态与监控放在右侧 inspector

## Success Criteria

- 第一次看 demo 的人能快速理解这不是静态卡片工具
- 第一次看 demo 的人能理解输入模态和训练形式都可以变化，但核心判断逻辑是一套
- 第一次看 demo 的人能理解科学复习系统与 agent memory 是两层能力，且复习系统不是附属装饰
- 第一次看 demo 的人能说清楚：这个系统面向的是“围绕真实项目持续学习”的任务，而不是普通陪聊问答
- 第一次看 demo 的人能从页面结构上直接看懂系统在安排学习：中间线程是动作与反馈，右侧 inspector 是状态与监控，而不是只听讲解才明白
- 团队成员能在不口头同步的情况下理解当前范围
- 任一新接手的人能在 10 分钟内找到项目规范、现状和下一步任务

## Current Implementation Target

- 维持一个简单 web demo
- 保留输入样例、学习者画像、状态、项目上下文和 agent 决策输出
- 当前默认主案例固定为“AI 工程师学习 RAG 系统设计并服务于手头项目”
- 强调“项目上下文 + 学习状态 + 动作选择”的统一模型，而不是泛化展示所有输入和学科
- 把科学复习放在系统中作为学习域能力，而不是产品本体
- 用受约束的单 pedagogical agent 先跑通编排闭环
- 第一版优先证明一次高价值判断：系统能区分用户是没懂、混淆了，还是已经懂了但需要巩固
- 第一版就接入 LangChain + LangGraph，但不追求复杂多 agent 和复杂 graph
- 当前前端目标是 activity-first：用户在同一线程里提问、随时附加材料，并在需要时通过轻量学习动作完成当前回合
- 用户侧默认不直出内部编排证据；中间线程优先展示动作与反馈，右侧 inspector 承载状态、材料摘要与监控信息
- 允许轻量 card deck、提交后短反馈和受约束输入区，但这些都只是候选学习形式的承载方式，不是产品目标本身
- 比赛版暂不实现真实多模态解析链路和完整 SRS 算法，但叙事上要保留“多模态输入 + 多类型学习 + SRS”这条长期主线

## Technical Direction

- `apps/web`: React + Tailwind + shadcn-friendly frontend
- `apps/agent`: Python + LangChain + LangGraph 的受约束单 agent core
- 两层之间通过显式 API contract 连接
- 第一版重点是 state、action、tool、guardrail 的设计，而不是复杂多 agent 编排
- graph、更多节点和更复杂 runtime 留作后续扩展，而不是第一版前提
