# Demo Modules Design

Date: 2026-04-21
Topic: 比赛答辩中的“功能介绍”与“核心 Agent 技术方案”模块

## Context

当前 demo 主链已经可讲，核心问题不再是补更多功能，而是用极短时间让评委听懂：

- Xidea 不是普通 AI tutor
- Xidea 不是项目资料整理或 AI summary 工具
- Xidea 的核心差异在于“学习编排 + 复习反馈驱动持续学习”

本设计文档只定义答辩中两个模块的讲法：

1. 功能介绍模块
2. 核心 Agent 技术方案模块

不在本文范围内：

- 全量比赛脚本
- 商业化叙事
- 逐页视觉设计细节
- 详细技术实现说明

## Audience And Constraints

- 主要载体：PPT
- 兼容载体：现场口播、录屏
- 功能介绍模块目标时长：约 1 分钟
- 技术方案模块目标：讲清核心 agent 脉络，不讲成框架清单或工程实现细节

## Module A: 功能介绍

### Design Goal

按用户旅程讲清系统如何持续组织学习，同时避免讲成功能列表。

### Core Message

- `Project` 是学习边界容器，不是项目管理壳
- 知识点才是系统编排的核心对象
- 系统会根据状态组织研讨、学习和复习
- 复习反馈会进入下一轮学习，而不是停留在当前轮次

### Recommended Structure

功能介绍模块固定为 5 页。

#### Page 1

标题：`为什么用 Project 组织学习`

要点：

- `Project` 用来约束学习主题、材料范围和持续状态
- 它不是单次聊天入口
- 系统先定义学习边界，再开始后续编排

#### Page 2

标题：`材料进入后，系统拆出知识点`

要点：

- 材料不会只被总结成一段话
- 系统会把内容转成可追踪、可学习、可复习的知识点对象
- 后续状态更新都围绕知识点展开

#### Page 3

标题：`为什么需要研讨 Session`

要点：

- 研讨不是闲聊，而是在继续澄清问题和边界
- 用户可以补材料、补上下文、修正理解
- 系统会借这个过程调整知识点，而不是只返回回答

#### Page 4

标题：`为什么学习和复习要分开`

要点：

- 学习针对“还没掌握”的知识点
- 复习针对“已经学过但需要巩固”的知识点
- 系统会根据当前知识状态，决定更适合进入哪种 Session

#### Page 5

标题：`复习反馈如何驱动持续学习`

要点：

- 学习和复习后的反馈，不会停留在当前这一轮
- 反馈会更新知识点状态与后续安排
- 下一次进入时，系统会基于这些反馈继续推进

### Presentation Rules

- 这 5 页按流程走，不按功能清单走
- 第 3 页只解释“为什么需要研讨”，不展开成聊天能力介绍
- 第 4 页只解释“为什么学习和复习分开”，不展开具体题型
- 第 5 页不用“内部写回”这种偏工程话术，统一表述为“复习反馈驱动持续学习”

### What To Avoid

- 把系统讲成“更聪明的 AI tutor”
- 把材料处理讲成“AI summary”
- 在这部分展开模型、LangGraph、API 或数据库
- 把三种 Session 讲成并列功能列表

## Module B: 核心 Agent 技术方案

### Design Goal

讲清 Xidea 的核心 agent 不是普通 chat + tools，而是围绕学习状态运行的闭环系统。

### Title

`核心 Agent 技术方案：用状态、决策、动作和写回形成学习编排闭环`

### Core Message

- agent 维护的不是一次聊天上下文，而是一组持续更新的学习状态
- agent 的核心价值不是直接回答，而是围绕当前状态做 pedagogical decision
- 本轮动作结果会进入后续依据，形成持续学习闭环

### Recommended Structure

技术方案模块固定为 3 页。

#### Page 1

标题：`状态 / 决策`

要点：

- agent 围绕 `知识点 + 学习状态 + 复习状态` 运行
- 它不是只看当前用户一句话，而是先读当前学习对象和状态
- 基于这些状态判断当前更适合研讨、学习还是复习

#### Page 2

标题：`动作执行`

要点：

- 决策完成后，agent 进入对应动作执行路径
- 研讨、学习、复习不是同一个回复模板，而是不同的动作类型
- tool / action 的作用是为当前学习决策服务，而不是通用任务执行

#### Page 3

标题：`写回闭环`

要点：

- 本轮学习或复习结果会写回知识点状态和后续安排
- 复习反馈会影响下一轮学习编排
- 所以系统不是一次性互动，而是持续学习系统

### Presentation Rules

- 主线固定为 `状态 -> 决策 -> 动作 -> 写回`
- 框架名只作为落地背书，不作为页面主线
- 如果需要提 LangChain / LangGraph，只能放在口播收尾一句，不占主结论

### What To Avoid

- 讲成通用 agent runtime 介绍
- 讲成 LangChain / LangGraph 教程
- 讲成“有 tools 所以很智能”
- 回到前面功能模块里已经讲过的产品对象介绍

## Relationship Between The Two Modules

- 功能介绍模块回答：系统对用户如何运作
- 技术方案模块回答：agent 在内部如何形成闭环
- 两个模块共享同一条主线：知识点是对象，状态决定动作，复习反馈驱动持续学习

## Suggested Speaking Priority

如果时间继续被压缩，优先保留：

1. 功能介绍模块的第 1、2、4、5 页主结论
2. 技术方案模块的第 1、3 页主结论

可以压缩但不应删除：

- 功能介绍里的研讨页
- 技术方案里的动作执行页

## Acceptance Check

本设计成立的标准是：

- 评委不会把 Xidea 误解成普通 AI tutor
- 评委不会把 Xidea 误解成 AI summary 工具
- 评委能复述“系统围绕知识点和状态持续组织学习”
- 技术方案部分能解释“为什么这不只是聊天 + tools”
